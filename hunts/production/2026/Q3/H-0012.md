## LEARN: Prepare the Hunt

### Hypothesis Statement

Adversaries use native log query utilities (wevtutil, Get-WinEvent, Get-EventLog on Windows; journalctl, ausearch, direct file access to /var/log/* on Linux) within 24 hours of initial access or privilege escalation to enumerate Windows Event Logs and Linux system logs, in order to identify monitoring blind spots, understand detection capabilities, and determine which logs must be cleared or avoided before further defense evasion. If true, we will observe log-enumeration process execution or file access events clustered in time shortly after a new logon session or privilege escalation event, disproportionately from non-administrative accounts or unusual parent processes, and in some cases followed by log-clearing activity (T1070.001).

### Threat Context

Log Enumeration (T1654) is adversary reconnaissance against defensive visibility: attackers query Windows Event Logs, Linux syslog/journald, and cloud audit logs (CloudTrail, Azure Activity Log, GCP Audit Log) to learn what is being recorded and where evidence of their intrusion might exist, before deciding what to clear (T1070.001) or avoid triggering. It commonly follows initial access or privilege escalation and precedes anti-forensic activity, making it a high-value early-warning signal. Relevant to this org's threat model (ransomware operators, insider threats, external attackers per environment.md) since all three actor classes have incentive to assess and evade logging before achieving their objective. See [research/R-0001.md](../../../../research/R-0001.md) for full research.

## OBSERVE: Expected Behaviors

### What Normal Looks Like

- System administrators using Event Viewer, `Get-WinEvent`, or `journalctl` for routine troubleshooting, tied to an open ticket
- SIEM/log-shipping agents (Splunk UF, Cortex XDR agent) continuously reading log files as part of normal collection
- Configuration management tools (Ansible, Puppet, Chef) checking logging service status during compliance runs
- Backup and log-rotation utilities (`logrotate`, scheduled backup scripts) accessing `/var/log/*`
- Security/compliance audit scanning tools performing scheduled log reviews
- IT service desk personnel investigating a specific user-reported issue (correlated with a ticket)

### What Suspicious Looks Like

- Log enumeration commands executed within ~5-60 minutes of a new interactive/remote logon (4624) or privilege escalation event (4672, sudo), with no corresponding ticket
- Enumeration from unusual parent processes (e.g., a web shell, scripting engine, or Office application spawning `wevtutil`/`powershell`/`journalctl`)
- Enumeration by non-administrative accounts or service accounts that don't normally touch logs
- Remote log enumeration via WMI, PowerShell Remoting, or SSH against a host the account doesn't normally administer (lateral movement context)
- Enumeration sequences immediately followed by log-clearing commands (`wevtutil cl`, `Clear-EventLog`, truncation/deletion of `/var/log/*`) - high-fidelity T1654 -> T1070.001 chain

### Expected Observables

- **Processes:** `wevtutil.exe` (`qe`, `el`, `gli` args), `powershell.exe`/`pwsh.exe` running `Get-WinEvent`/`Get-EventLog`, `journalctl`, `ausearch`, `lastlog`, `cat`/`tail`/`grep`/`less` targeting `/var/log/*`
- **Network:** Not primary for enumeration itself; check for concurrent WinRM (5985/5986) or SSH (22) sessions indicating remote log queries during lateral movement
- **Files:** Access to `C:\Windows\System32\winevt\Logs\*.evtx`, `/var/log/*`, `/var/log/auth.log`, `/var/log/audit/audit.log`
- **Registry:** Queries to `HKLM\SYSTEM\CurrentControlSet\Services\EventLog\*` (event log configuration/size checks) - limited telemetry per R-0001
- **Authentication:** Temporal proximity to 4624 (logon), 4672 (privileged logon assigned), or Linux `sudo`/`su` events immediately preceding enumeration

#### Step 1: Baseline Count - Windows log enumeration (Cortex XDR / XQL)

```xql
dataset = xdr_data
| filter event_type = ENUM.PROCESS and event_sub_type = ENUM.PROCESS_START
| filter action_process_image_name in ("wevtutil.exe", "powershell.exe", "pwsh.exe")
| filter action_process_image_command_line contains "wevtutil"
   or action_process_image_command_line contains "Get-WinEvent"
   or action_process_image_command_line contains "Get-EventLog"
| comp count() as event_count
```
