## 1. System Research: How It Works

### Summary
Log enumeration is the process of discovering, accessing, and cataloging system logs to understand system activity, diagnose issues, or identify security events. Security researchers and administrators use various tools and techniques to locate logs across different operating systems (Windows Event Logs, Linux syslog/journald, application logs) and analyze their contents for patterns, anomalies, or evidence of compromise.

### Key Findings
- Different operating systems store logs in standard locations: Windows uses Event Viewer and stores logs in %SystemRoot%/System32/winevt/Logs/, while Linux systems typically use /var/log/ directory with files like syslog, auth.log, and systemd journal
- Log enumeration tools include native utilities (wevtutil, Get-EventLog, journalctl, grep) and third-party solutions that can query, filter, and export logs based on time ranges, event IDs, severity levels, or keywords
- Logs contain valuable security information including authentication attempts, privilege escalations, process execution, network connections, and system changes that are critical for incident response and threat hunting
- Access to logs typically requires elevated privileges (administrator/root), and log files themselves may be protected by access controls to prevent tampering or unauthorized disclosure
- Log rotation and retention policies affect enumeration scope, as older logs may be archived, compressed, or deleted based on system configuration and storage constraints

## 2. Adversary Tradecraft: Attack Techniques

### Summary
Adversaries enumerate system and application logs to identify logging capabilities, understand what activities are being recorded, and determine where evidence of their intrusion may exist. This reconnaissance helps attackers identify which logs to clear, disable, or avoid triggering during their operations to evade detection and maintain persistence.

### Key Findings
- Attackers commonly target Windows Event Logs (Security, System, Application) using built-in tools like wevtutil.exe, Get-WinEvent PowerShell cmdlet, and querying the registry at HKLM\SYSTEM\CurrentControlSet\Services\EventLog to identify logging configurations
- On Linux systems, adversaries enumerate logs in /var/log/ directory, check syslog configurations (/etc/rsyslog.conf, /etc/syslog.conf), and review systemd journal settings to understand logging scope and retention
- Cloud environment log enumeration includes querying AWS CloudTrail status, Azure Activity Logs, and GCP Cloud Audit Logs through CLI tools or APIs to assess monitoring coverage before executing malicious activities
- Adversaries look for security tool logs (EDR, SIEM, IDS/IPS) to understand detection capabilities and identify blind spots where their activities may go unnoticed
- Common indicators include unusual queries to logging service APIs, access to log configuration files by non-administrative processes, execution of log enumeration commands (wevtutil gli, Get-EventLog, ls /var/log), and registry queries to EventLog keys
- This technique often precedes log clearing (T1070.001) or indicator removal activities, making log enumeration detection critical for early warning of anti-forensic activities

## 3. Telemetry Mapping: OCSF Fields

### Summary
Log Enumeration (T1654) involves adversaries querying, accessing, or enumerating system logs to understand security monitoring, identify evidence of their activities, or plan evasion techniques. Telemetry would capture file access to log directories, execution of log query utilities (wevtutil, Get-WinEvent, journalctl, ausearch), API calls to event log services, and process command lines indicating log enumeration activity. Windows Security Event Logs, Sysmon process creation events, and EDR telemetry provide primary detection opportunities.

### Key Fields
- activity_id (95% populated): Set to 1 (Open) or 2 (Read) when adversary accesses log files; critical for detecting unauthorized log file access patterns
- process.cmd_line (90% populated): Captures commands like 'wevtutil qe Security', 'Get-EventLog -LogName Security', 'journalctl --list-boots', 'cat /var/log/auth.log' indicating log enumeration attempts
- process.file.name (98% populated): Identifies log enumeration tools such as wevtutil.exe, powershell.exe, wmic.exe, auditctl, or journalctl that are commonly used to query logs
- file.path (85% populated): Reveals access to log locations like C:\Windows\System32\winevt\Logs\*.evtx, /var/log/*, /var/log/audit/audit.log, or application-specific log directories
- actor.process.name (92% populated): Parent process context (e.g., cmd.exe, powershell.exe, bash) helps identify whether log enumeration is part of scripted reconnaissance or manual exploration
- api.operation (70% populated): Windows API calls like OpenEventLog, ReadEventLog, or EvtQuery indicate programmatic log access that may bypass command-line detection

## 5. Research Synthesis

### Executive Summary
Log Enumeration (T1654) is a reconnaissance technique where adversaries query system, application, and security logs to understand monitoring coverage, identify evidence of their activities, and plan evasion strategies. Attackers leverage native OS utilities (wevtutil, Get-EventLog, journalctl) and direct file access to enumerate Windows Event Logs, Linux syslog/journald, and cloud audit logs across AWS, Azure, and GCP environments. Detection opportunities exist through process command-line monitoring, file access telemetry to log directories, and API calls to event log services, though a related hunt (H-0012) already provides 78% coverage overlap.

### Recommended Hypothesis
> Adversaries use native log query utilities and direct file access to enumerate system and security logs on Windows, Linux, and cloud platforms to identify monitoring blind spots, understand detection capabilities, and determine which logs must be cleared or avoided to maintain operational security.

### Gaps Identified
- Cloud-native log enumeration telemetry coverage is underspecified - while adversary tradecraft mentions AWS CloudTrail, Azure Activity Logs, and GCP Cloud Audit Logs enumeration via CLI/API, the telemetry mapping focuses primarily on traditional OS-level indicators without detailing cloud API calls, SDK usage patterns, or SaaS application log access methods.
- Correlation between log enumeration and subsequent evasion actions is not addressed - detection of enumeration alone may generate false positives from legitimate administrative activity, but linking enumeration to log tampering, clearing, or disabling within temporal proximity would provide higher-fidelity detection logic.

### Key Findings
- Hypothesis: Adversaries use native log query utilities and direct file access to enumerate system and security logs on Windows, Linux, and cloud platforms to identify monitoring blind spots, understand detection capabilities, and determine which logs must be cleared or avoided to maintain operational security.
- Gap: Cloud-native log enumeration telemetry coverage is underspecified - while adversary tradecraft mentions AWS CloudTrail, Azure Activity Logs, and GCP Cloud Audit Logs enumeration via CLI/API, the telemetry mapping focuses primarily on traditional OS-level indicators without detailing cloud API calls, SDK usage patterns, or SaaS application log access methods.
- Gap: Correlation between log enumeration and subsequent evasion actions is not addressed - detection of enumeration alone may generate false positives from legitimate administrative activity, but linking enumeration to log tampering, clearing, or disabling within temporal proximity would provide higher-fidelity detection logic.
- Focus: Prioritize detection of log enumeration from non-administrative contexts and unusual user accounts, particularly focusing on lateral movement scenarios where compromised accounts query logs on remote systems via WMI, PowerShell remoting, or SSH, as this indicates advanced reconnaissance rather than routine administration.
- Focus: Develop behavioral baselines for legitimate log access patterns by IT operations and security teams to enable anomaly detection, focusing on frequency, volume, time-of-day patterns, and specific log types accessed to differentiate adversary reconnaissance from normal administrative activity.
- Focus: Investigate the 78% overlap with existing hunt H-0012 to identify specific gaps in that coverage, then focus new hunt development on underrepresented areas such as application-specific logs (database, web server, security tool logs), container/orchestration logs (Docker, Kubernetes), and cross-platform enumeration sequences that span multiple log sources.
